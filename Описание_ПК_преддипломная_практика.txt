================================================================================
ФОРМАЛЬНОЕ ОПИСАНИЕ ПРОФЕССИОНАЛЬНЫХ КОМПЕТЕНЦИЙ (ПК)
для отчёта по преддипломной практике
(проект: информационная система электронной библиотеки, веб‑клиент, REST API,
мобильный клиент; СУБД PostgreSQL; платформа ASP.NET Core; ORM Entity Framework Core)
================================================================================

В ходе преддипломной практики работа выполнялась поэтапно, в логике жизненного цикла
базы данных и прикладной информационной системы. Основное внимание уделено анализу
предметной области, проектированию структуры БД, её реализации в СУБД PostgreSQL,
разработке и сопровождению веб‑приложения LibraryMPT, программного интерфейса
REST API и мобильного клиента на платформе Flutter, а также настройке механизмов
администрирования и защиты данных. Ниже приведено формальное описание освоенных
профессиональных компетенций (ПК) в соответствии с перечнем практики.

Стиль изложения: официально‑деловой; без использования первого лица множественного
и единственного числа в отношении автора отчёта; фиксируются выполненные действия,
результаты и используемые средства. Объём по каждой ПК: 3–5 предложений.

Нумерация рисунков ниже — сквозная по тексту файла; при вставке в отчёт с другими
разделами нумерацию и формулировки подписей следует привести к требованиям
методических указаний и ГОСТ (при необходимости — перенумеровать, объединить
или разбить составные снимки на отдельные рисунки).

ПМ.11. Разработка, администрирование и защита баз данных
--------------------------------------------------------------------------------

ПК 11.1. Осуществлять сбор, обработку и анализ информации для проектирования баз данных.
В ходе практики выполнен сбор сведений о сущностях предметной области (пользователи,
роли, каталог книг, подписки, заявки, журналы действий) на основе существующей
веб‑части приложения и требований к мобильному доступу. Проведена систематизация
полей и связей по моделям домена и по фактической структуре таблиц в СУБД
PostgreSQL. Составлена сводная таблица соответствия сущностей таблицам и полям
СУБД, зафиксированы первичные и внешние ключи, типы данных и задействованные
представления; материалы включены в сопровождение практики. Этап выполнялся
в первые 1–2 недели практики; использованы документация проекта, pgAdmin,
средства IDE и построение диаграмм связей (в том числе в draw.io).

  Рисунок 2 — Фрагмент сводной таблицы соответствия «сущность — таблица — поле»
              или снимок обозревателя таблиц в pgAdmin.

ПК 11.2. Проектировать базу данных на основе анализа предметной области.
На основании анализа выполнено уточнение логической модели: выделены сущности,
атрибуты, связи «многие‑ко‑многим» (в том числе автор–книга), роли доступа.
Спроектированы дополнения к модели данных под новые сценарии (прогресс чтения,
заявки на книги, расширения API) с учётом ограничений целостности и типовых
выборок каталога. Задокументированы предполагаемые индексы и ограничения внешних
ключей; решения согласованы с руководителем практики в объёме задания.

Рисунки к ПК 11.2:

              ограничений внешних ключей.

ПК 11.3. Разрабатывать объекты базы данных в соответствии с результатами анализа
предметной области.
Подготовлены и скорректированы объекты БД: таблицы, ограничения, при необходимости
представления или триггеры в объёме, предусмотренном заданием практики. Изменения
отражены в скриптах миграций или в пакете изменений для администратора БД.
Проведена проверка согласованности имён столбцов с маппингом в Entity Framework
Core (настройка Fluent API в DbContext); при несовпадении соглашений об именовании
столбцов выполнена доработка конфигурации модели и повторный прогон миграций.
Объём доработок распределён на 2–3 итерации по согласованию с руководителем.

Рисунки к ПК 11.3:

ПК 11.4. Реализовывать базу данных в конкретной системе управления базами данных.
База данных развёрнута в СУБД PostgreSQL на локальной машине и/или учебном стенде:
создан экземпляр БД, настроена строка подключения в конфигурации приложения,
выполнены миграции либо импорт схемы. Проверена работоспособность подключения
из ASP.NET Core через провайдер Npgsql, выполнены тестовые обращения к данным
через контекст EF Core. Зафиксированы версия СУБД и параметры окружения (порт,
кодировка) в описании стенда.

ПК 11.5. Администрировать базы данных.
Осуществлены резервное копирование и восстановление в учебном объёме (pg_dump,
восстановление из дампа), назначены права роли приложения с минимально необходимым
набором привилегий SELECT/INSERT/UPDATE на рабочие таблицы. Проведён контроль
объёма журнальных таблиц (audit); на стенде выполнена очистка тестовых данных
по согласованию. Для передачи стенда подготовлен перечень выполненных операций
и параметров подключения (без раскрытия секретов в открытом виде).

Рисунки к ПК 11.5:

  Рисунок 10 — Фрагмент журнала или консоли: успешное выполнение резервного
               копирования (pg_dump) либо восстановления из дампа.

ПК 11.6. Защищать информацию в базе данных с использованием технологии защиты
информации.
Реализованы меры на уровне приложения и данных: пароли хранятся в виде
криптографического хэша, доступ к данным осуществляется через API с
аутентификацией JWT, в ответах используются DTO без избыточного набора полей.
Секреты вынесены из репозитория в User Secrets и/или переменные окружения на
стенде. Запросы к БД формируются через EF Core с параметризацией; проведён разбор
типовой угрозы SQL‑инъекции и показано снижение риска за счёт выбранного подхода.
